# tcpquic-proxy 后续完善 TODO（2026-07-02）

本文基于当前仓库源码、文档、测试脚本和静态分析结果整理，按优先级列出后续可继续完善的方向。

## P0：安全与稳定性

### 1. 收敛生产安全默认值

- `src/config/config.h` 当前默认 `QuicDisable1RttEncryption{true}`，文档也记录该默认更偏 lab 配置。
- 建议把生产默认改为启用 1-RTT 加密，或至少引入明确的 `--lab-disable-1rtt-encryption` 语义。
- release 测试矩阵必须覆盖显式 encrypted 1-RTT mode。
- `README.md` 中 `--allow-targets` 默认 `0.0.0.0/0` 适合开发但不适合生产。建议增加 production profile、启动警告，或在生产配置中强制显式 ACL。

### 2. 移除生产路径中的 `abort()`

三套 relay worker 遇到 MsQuic “FIN-only receive without known final size” 时会直接终止进程：

- `src/tunnel/linux_relay_worker.cpp`
- `src/tunnel/darwin_relay_worker.cpp`
- `src/tunnel/windows_relay_worker.cpp`

建议改成 per-tunnel fail/close + 指标 + trace，避免异常流量或边界事件打掉整个代理进程。

### 3. Windows FIN / receive 顺序问题专项闭环

- `docs/bugs/iocp-fin-out-of-order.md` 已记录 Windows receive view 可能越过 FIN 的怀疑。
- 建议先增加 `ReceiveSeq`、`FinReceiveSeq`、`AbsoluteOffset` 日志与断言。
- 再修复 worker 队列顺序或 FIN barrier。
- 该问题影响 Windows 下载方向稳定性，优先级较高。

## P1：质量门禁与回归能力

### 4. 建立 CI / CTest 入口

- CMake 已定义大量 `tcpquic_*_test`，但当前未见 `enable_testing()` / `add_test()` 注册。
- 仓库也未命中明显 CI 配置。
- 建议补齐 `ctest` 聚合目标、Linux Release / ASAN 两套 CI、最小脚本 smoke。
- 目标是避免依赖手工执行 `for t in build/bin/Release/tcpquic_*_test; do "$t"; done`。

### 5. 把关键脚本测试产品化为可重复 gate

现有脚本覆盖不错：

- `scripts/test-tcpquic-proxy.sh`
- `scripts/test-tcpquic-concurrent.sh`

建议拆成 smoke / integration / stress 分层，并统一：

- 超时策略
- 日志归档
- 端口冲突处理
- CI 入口
- 失败时关键日志截取

### 6. 补多网口与热更新回归

- `docs/test/dgx-multi-interface-quic-binding-execution-20260628_cn.md` 已写明需要增加同端口多 listen bind plan 集成测试。
- 还需要补 admin PATCH 修改 paths 的 case。
- 多路径是项目核心能力，建议纳入高优先级回归。

## P2：可观测性与运维能力

### 7. 补齐 Admin API 覆盖

`docs/admin-api/interface.md` 已列出当前缺口，建议继续补齐：

- per-peer proto 字段
- proxy auth
- connection stream count
- keepalive
- handshake threads
- diagnostics 动态开关
- 真实 worker relay 明细

建议先做只读完整性，再设计可变更接口和失败回滚语义。

### 8. relay worker snapshot 与低扰动诊断

Windows、macOS、Linux relay 文档都提到 snapshot、锁竞争、统计原子和 event queue 观测成本。

建议优先把 per-worker 快照做成低频/按需采样，避免 Admin 高频查询反过来干扰数据面。

## P3：性能与跨平台完善

### 9. 高 RTT / loss 下 MsQuic batch 参数可配置化评估

- `docs/io-throughput-diag_cn.md` 记录 `QUIC_MAX_DATAGRAMS_PER_SEND=240` 是否需要做成构建/运行配置仍未决。
- 建议先做构建开关或 patch 管理说明。
- 若后续需要跨环境部署，再评估是否暴露为运行时配置。

### 10. 补齐 macOS 实机验证

- `docs/relay_macos.md` 明确当前 Linux 环境无法验证 Darwin kqueue / MsQuic callback 组合。
- 建议在 macOS CI 或专用机器跑 relay unit / IO tests。
- 重点覆盖 late callback、terminal callback 后不再提交新的 TCP->QUIC send、callback 高并发预算一致性等场景。

### 11. Windows callback 锁热点先观测后优化

- `docs/relay_win.md` 已分析 `Lock_` 热点风险。
- 建议先加 callback 到 IOCP post 耗时、`FindRelayById()` 次数、锁等待时间等观测。
- 再决定是否做 weak binding pin relay 或纯投递 callback 路径。

## P4：仓库与发布卫生

### 12. 测试证书和生成物治理

- `cert/*.key` 已随仓库提交，文档说明仅限测试。
- 建议在发布包和容器构建中明确排除测试私钥。
- README 的生产部署路径应避免引用这些测试 key。
- 当前工作区还有 untracked `__pycache__`、空 `cfg.json`、`.cursor/.claude` 等生成或本地配置文件。建议更新 `.gitignore` 或清理生成物，降低提交噪音。

## 补充观察

- 当前源码体量约 4.2 万行，最大生产文件集中在 `linux_relay_worker.cpp`、`windows_relay_worker.cpp`、`darwin_relay_worker.cpp`、`tcp_tunnel.cpp`、`quic_session.cpp`。
- 这些模块也是高并发、跨线程、平台差异和性能调优最敏感的区域，后续改动应优先配套专项回归和诊断指标。
- 通用 SAST / unsafe pattern 扫描未发现明显高危项，但这不能替代高并发数据面和跨平台实机测试。
