# Built-in Speed Test

## Goal

The built-in speed test measures authenticated end-to-end TCP-over-QUIC tunnel throughput without an external HTTP server, curl target, or iperf process.

The client opens a QUIC control stream to the server. The server creates a temporary loopback TCP listener and returns that port. The client then opens normal tunnel streams to that temporary loopback target and pumps generated bytes through local socket pairs.

## Commands

Start the server:

```bash
./build/bin/Release/tcpquic-proxy server \
  --quic-listen 127.0.0.1:14443 \
  --quic-cert cert/server/server.crt \
  --quic-key cert/server/server.key \
  --quic-ca cert/ca.crt \
  --allow-targets 10.0.0.0/8
```

Run a download test:

```bash
./build/bin/Release/tcpquic-proxy client \
  --quic-peer 127.0.0.1:14443 \
  --quic-cert cert/client/client.crt \
  --quic-key cert/client/client.key \
  --quic-ca cert/ca.crt \
  --quic-connections 1 \
  --compress off \
  --download-test 10
```

Run an upload test:

```bash
./build/bin/Release/tcpquic-proxy client \
  --quic-peer 127.0.0.1:14443 \
  --quic-cert cert/client/client.crt \
  --quic-key cert/client/client.key \
  --quic-ca cert/ca.crt \
  --quic-connections 1 \
  --compress off \
  --upload-test 10
```

For multi-connection testing, increase `--quic-connections`. The client creates one data tunnel per configured QUIC connection.

## Result Fields

Example output:

```text
speed-test upload: local_bytes=2119958528 server_bytes=2119958528 local_seconds=1.000 server_seconds=1.008 gbps=16.833 mib_s=2006.60 accepted=1 closed=1
```

Fields:

| Field | Meaning |
|-------|---------|
| `local_bytes` | Bytes generated or consumed by the client-side pump worker. |
| `server_bytes` | Bytes generated or consumed by the server-side temporary TCP worker. |
| `local_seconds` | Requested client test duration. |
| `server_seconds` | Server session lifetime from `SPEED_START` to `SPEED_FINISH` completion. |
| `gbps` / `mib_s` | Primary throughput for the tested direction. |
| `accepted` / `closed` | Number of temporary server TCP connections accepted and closed. Should match `--quic-connections`. |

Throughput source:

- Upload uses `server_bytes` and `server_seconds`, because local socket writes can briefly run ahead of bytes that actually traverse the tunnel.
- Download uses client `local_bytes` and `local_seconds`, because the server may generate extra bytes that are still buffered when the client stops the timed read.

Upload and download fail when local/server byte counts differ by more than 1% or 16 MiB. Download still reports throughput from client-drained bytes because the server may generate a small amount of extra data before `SPEED_FINISH` stops the temporary TCP worker.

## Compression Behavior

For IO throughput tests, use `--compress off`.

When the global client config is left at the normal default `--compress auto`, the speed-test data tunnels are forced to `off`. This avoids measuring compression policy bootstrap behavior instead of tunnel IO throughput. It also keeps the default local smoke commands stable, because `auto` has no prior compression sample and would otherwise choose lz4 for the first stream.

Explicit compression is still honored:

```bash
./build/bin/Release/tcpquic-proxy client ... --compress lz4 --download-test 10
./build/bin/Release/tcpquic-proxy client ... --compress zstd --download-test 10
```

Current caveat: explicit compressed speed tests are useful for debugging the compression path, but they are not the recommended baseline for IO throughput. Multi-connection lz4 download has shown unstable `SPEED_RESULT` completion in local smoke testing and should not be used as the standard pass/fail benchmark until the compression relay FIN path is separately hardened.

## Platform Scope

The built-in speed-test feature is designed to be cross-platform, not Linux-only.

The command-line options, control protocol, client runner, server controller, and tunnel data path are shared by Linux and Windows. Local TCP operations go through the project socket abstraction:

- `TqSocketHandle`
- `TqSocketPair`
- `TqSetSocketBuffer`
- `TqGetSocketBuffer`
- `TqSetNoDelay`
- `TqShutdownSend` / `TqShutdownBoth`

`TqSocketPair` has both POSIX and Windows implementations in `src/platform/platform_socket_posix.cpp` and `src/platform/platform_socket_win.cpp`. The speed-test timeout helper also has separate Windows and POSIX branches: Windows uses `DWORD` socket timeouts, while POSIX uses `timeval`.

The 2026-06-13 DGX performance fix and 16-17 Gbps validation are Linux-specific measurements. That fix makes the built-in loopback producer and client socketpair endpoints use the active TCP throughput buffer instead of a hard-coded 256 KiB buffer. On Linux, `getsockopt(SO_SNDBUF/SO_RCVBUF)` commonly reports roughly twice the requested value, so byte-count mismatch tolerance accounts for high-throughput socket pipeline depth.

Windows should compile through the same abstraction, but Windows throughput and stop-boundary byte-count behavior have not been validated with the same DGX-style test matrix. Treat Linux/DGX throughput numbers as Linux results until Windows has its own build and upload/download verification.

## ACL Bypass Scope

The server-side temporary listener is reachable only through the authenticated speed-test control flow. Normal user tunnels still use the configured ACL.

The ephemeral bypass is scoped to:

- exact active speed-test session port
- loopback target returned by `SPEED_READY`
- lifetime before `SPEED_FINISH` or server shutdown cleanup

This allows local tests with an ACL such as `--allow-targets 10.0.0.0/8` even though the temporary target is `127.0.0.1`.

## Operational Notes

- `--download-test` and `--upload-test` are client-only and mutually exclusive.
- Speed test mode is only supported in single-peer client mode. It is rejected with router `--client-config`.
- Speed test mode is rejected together with `--warmup-mb`; both features need to own startup traffic.
- The generated payload is deterministic pseudo-random data to avoid accidentally turning IO tests into compression-ratio tests.
- Server accepted sockets use short send/receive timeouts so `SPEED_FINISH` can stop the temporary workers reliably even when TCP send is blocked.
