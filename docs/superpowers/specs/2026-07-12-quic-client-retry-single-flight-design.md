# QUIC 客户端固定间隔单飞重连设计

## 背景

QUIC 客户端连接失败时，当前实现按固定 3000ms 间隔调度重连。每个连接槽位使用共享布尔值 `RetryScheduled` 表示已有重试，但延迟任务本身没有唯一身份。旧连接关闭完成、当前连接启动以及新一轮调度会重复修改同一个布尔值，使延迟队列中已经过期的任务可能被后续调度重新激活。

结果是同一槽位逐渐积累多个可执行重试任务。服务器持续不可达时，重连频率会从每 3 秒一次增长为每秒多次，造成短命 QUIC 连接风暴、高 CPU 占用和大量日志输出。

## 目标

- 保持每个连接槽位固定每 3000ms 重试一次，不引入指数退避。
- 每个槽位任意时刻最多存在一个有效、可执行的重试任务。
- 已被取代或取消的历史任务即使仍留在 ingress 延迟队列中，也必须永久失效。
- 旧连接的异步关闭完成不能改变当前连接或当前重试的状态。
- 保持多槽位相互独立：不同槽位可以各自拥有一个有效重试任务。

## 非目标

- 不修改重试间隔配置方式。
- 不增加延迟任务物理取消接口。
- 不改变 QUIC 握手、路径选择、连接负载均衡或日志策略。
- 不处理本地 SOCKS/HTTP 监听端口冲突。

## 方案选择

### 采用：槽位级重试序号

每个 `ClientSlot` 增加单调递增的重试序号。创建延迟任务时捕获本轮序号；任务执行前必须验证捕获值仍等于槽位当前值。任何取消或替换当前重试的操作都会推进序号，因此旧任务无法被后续共享状态重新激活。

这种方案只修改 QUIC session 内部状态，不要求 ingress reactor 支持删除任意延迟任务，也不复用具有连接身份语义的现有 `Generation`。

### 未采用：延迟任务取消句柄

让 `EnqueueDelayed()` 返回句柄，并在重新调度时从队列删除旧任务，可以物理消除历史任务，但会扩大 ingress reactor 接口和生命周期管理范围。本问题只要求保证任务不可执行，不需要物理删除，因而不采用。

### 未采用：复用连接 Generation

现有 `Generation` 用于手动重连、连接身份和终止升级校验。自动重试序号与连接身份的生命周期不同，复用会使管理面语义和自动恢复机制耦合，因而不采用。

## 状态模型

每个客户端槽位维护：

- `RetryScheduled`：当前是否存在一个有效重试。
- `RetryGeneration`：当前有效重试的身份序号；只增不减。
- `Connection` 和 `Context`：当前发布到槽位的连接及回调上下文。
- `Generation`：现有连接身份代次，语义保持不变。

需要作废当前重试时，必须在持有 `ClientSharedState::Lock` 的情况下：

1. 将 `RetryScheduled` 设为 `false`。
2. 递增 `RetryGeneration`。

递增序号是失效保证的核心。仅清除布尔值不够，因为后续新调度会再次将布尔值设为 `true`。

## 调度流程

`ScheduleStartRetry(index)` 在锁内执行以下操作：

1. 验证 session 正在运行且槽位存在。
2. 若 `RetryScheduled` 已为 `true`，不创建新任务。
3. 获取 scheduler 和 session gate。
4. 递增槽位的 `RetryGeneration`，把新值记录为本轮重试序号。
5. 设置 `RetryScheduled=true`。
6. 以固定 3000ms 延迟提交捕获本轮序号的任务。

如果提交失败，只能在槽位仍指向本轮序号时清除状态并推进序号，避免失败清理覆盖并发产生的新调度。

## 延迟任务执行流程

延迟任务在锁内同时验证：

- session 仍处于 started 且非 stopping 状态；
- 槽位索引仍有效；
- `RetryScheduled` 为 `true`；
- 槽位当前 `RetryGeneration` 等于任务捕获的序号。

任一条件不满足，任务直接返回。全部满足时，任务先清除当前有效重试状态并推进序号，再通过 session gate 调用一次 `StartSlot(index)`。

因此，历史任务不能借用新一轮调度设置的 `RetryScheduled=true` 重新获得执行资格。

## 连接关闭与重启流程

`SHUTDOWN_COMPLETE` 可能来自当前连接，也可能来自已经移出槽位的旧连接。回调只有在以下身份均仍匹配时，才可以清理槽位并安排下一轮重试：

- 槽位的 `Context` 等于回调的 `slotContext`；
- 槽位的 `Connection` 等于回调的 connection；
- 槽位连接 `Generation` 等于回调捕获的 generation。

若身份不匹配，该连接按 orphaned connection 完成自身关闭和资源释放，但不得清除当前槽位的 `RetryScheduled`，也不得调用 `RestartSlotAfterShutdownComplete()`。

若身份匹配，则清理当前连接状态、作废旧重试身份，并调用 `ScheduleStartRetry()` 创建唯一的下一轮 3000ms 重试。

## 其他失效路径

以下路径必须使用相同的槽位级失效操作：

- session stop；
- 删除或缩减连接槽位；
- 管理接口触发手动 reconnect；
- `StartSlot()` 接管某个有效重试并准备启动连接；
- 当前连接成功建立后清除不再需要的重试状态；
- 延迟任务提交失败。

这些路径不能只写 `RetryScheduled=false`。

## 并发与锁约束

- `RetryScheduled` 和 `RetryGeneration` 的读写均受 `ClientSharedState::Lock` 保护。
- 延迟任务不得在持锁时调用 `StartSlot()`，避免递归加锁。
- 调度器调用继续在锁外执行，避免 ingress reactor 回调与 session 锁形成锁顺序反转。
- 调度提交失败后的回滚必须校验捕获的重试序号，不能覆盖更新的状态。
- 不改变不同槽位之间的并发关系。

## 测试设计

在 `src/unittest/quic_session_reconnect_test.cpp` 增加确定性单元测试，使用现有可注入 scheduler 和 reconnect test hooks 保存任务并控制执行顺序。

覆盖以下行为：

1. 对同一槽位重复调度只提交一个 3000ms 任务。
2. 第一轮任务被作废、第二轮任务建立后，先执行第一轮任务不会调用 `StartSlot()`，第二轮任务只调用一次。
3. 旧连接的 shutdown complete 身份与当前槽位不匹配时，不清除当前重试，也不增加调度数量。
4. 当前连接失败关闭后只生成一个下一轮任务；重复关闭通知不产生额外有效任务。
5. stop 后执行保存的任务不会启动连接。
6. 手动 reconnect 或连接 generation 改变后，旧任务不会启动新连接。
7. 多槽位分别调度时，每个槽位各有一个有效任务，互不取消。
8. 现有固定延迟测试继续验证延迟严格等于 3000ms。

回归测试必须先在未修复代码上因旧任务错误执行或重复调度而失败，再实施最小生产代码修改使其通过。

## 验收标准

- 所有新增重连回归测试通过。
- 现有 `tcpquic_quic_session_reconnect_test` 全部通过。
- 相关项目测试与 Release 构建通过。
- 在持续连接失败场景中，单槽位的连接尝试间隔保持约 3000ms，不随运行时间缩短。
- 同一槽位不会因为 orphaned 连接关闭或历史延迟任务执行而产生第二个有效重试。
